What is First Time Liability (FTL)? NEW

First Time Liability occurs when a customer becomes liable for land tax for the first time - they had NO prior assessment for that property in the retrospective period (current + 4 prior years).

```

Four FTL Sources

  • Programmatic Creation: Automated data cleansing identifies new liability
  • Data Matching: Address/ownership reconciliation finds missing record
  • Valuation Change: Property value exceeds exemption threshold
  • Staff-Initiated: Customer reports new ownership or missing assessment

NOT FTL (Common Errors)

  • Address change only (existing liability, address updated)
  • PPR exemption removal (existing liability, exemption changed)
  • Liability recalculation (existing liability being amended)
  • Annual renewal (existing liability continuing)
Input data: Customer record, property history, assessment data
Processing responsibility: Staff (FTL identification)
Decision: Is this first liability OR existing liability change?
Output: FTL confirmed → proceed to Q0; or non-FTL process
```

Different FTL sources require different validation. Identify source type first:

Which FTL source? (Programmatic / Data Match / Valuation / Staff-Initiated)

Source type determines validation requirements. Document source in action item.
Input data: FTL action item source details
Processing responsibility: Staff (source identification)
Decision: Which FTL source type (4 categories)?
Output: Confirmed source → Q1 uses source-specific validation

Q1 - Has FTL Been Correctly Identified? IMPROVED

  • Customer has NO prior assessment for property in retrospective period
  • Property identified is correct (V/F, Crown Allotment)
  • Customer/ownership match is correct
  • Perform source-specific validation (based on Q0)
If ANY check fails, reject FTL. Do not proceed with invalid FTL.
Input data: FTL action item, prior assessment history, FTL source
Processing responsibility: Staff (4-point validation + source checks)
Decision: Is FTL valid and correct?
Output: Valid → Q2; Invalid → Reject and close

Q2 - Is FTL Item Ready to Process? IMPROVED

  1. Launch FTL Investigations item in e-Sys workflow
  2. Allocate to yourself; set status to "In Progress"
  3. Load related customers (if joint ownership)
  4. Check for conflicting action items on property/customer
  5. If conflicts found → Redirect to LOAN Complex tray
  6. If no conflicts → Proceed to Q3
Conflicting items: Active investigations, disputes, pending amendments, multiple FTLs on same property
Input data: FTL item status, related customers, action item list
Processing responsibility: Staff (item allocation, conflict check)
Decision: Ready for processing OR redirect to complex tray?
Output: Processing proceeds OR escalate to LOAN Complex

Q3 - Is Customer Under Investigation? IMPROVED

  1. Check for red exclamation mark (!) on customer record
  2. Check "Under Investigation" non-issue indicator
  3. Check related customers (joint owners) for investigation flags
Investigation Active?
• YES → Notify Investigator; redirect FTL item; pause processing
• NO → Proceed to Q4
Input data: Investigation indicators, customer flags
Processing responsibility: Staff (investigation check)
Decision: Investigation active YES/NO?
Output: Active → Redirect; Inactive → Continue to Q4

Q4 - Determine Assessment Correspondence Address IMPROVED

When sources conflict, apply hierarchy priority:

Priority Source Rule
1 RTBA Most recent (if property rented)
2 ATO Rental Current rental address
3 DLS Most recent address on record
4 COI Ownership address (corroborated)
5 NOA Customer-reported address
6 e-Sys Current e-Sys address (default)
Use MOST RECENT address from highest-priority source. Document which source used.
Input data: Address data from 6 sources
Processing responsibility: Staff (hierarchy selection)
Decision: Which address to use (hierarchy priority)?
Output: Assessment correspondence address confirmed

Q5 - Verify & Apply Principal Place of Residence (PPR) IMPROVED

PPR Verification Sources

  • Property Classification: Residential property (not commercial/industrial)
  • RTBA Data: Property address matches customer's rental bond address
  • ATO Rental Data: Customer in ATO system as tenant at property
  • Landata/Title: Property is residential land
  • Customer Declaration: Customer claim of PPR use
```
PPR Decision Logic:
• Residential + Occupancy + No exemption → Apply PPR
• Non-residential OR Not occupied → Do NOT apply
• Uncertain → Escalate to Senior Officer
Input data: Property classification, occupancy indicators
Processing responsibility: Staff (verification checks)
Decision: Apply / Amend / Remove PPR?
Output: PPR status set; liability impact calculated
```

Q6 - Resolve Duplicate Customers IMPROVED

Valid vs. Invalid Duplicates

  • Valid Duplicate: Same person with two records (data error to correct)
  • Invalid Duplicate: Different people with same/similar name (leave separate)
```

Decision Criteria

  • Same date of birth? → Likely valid
  • Same address? → Likely valid
  • Same ID (ABN, ACN)? → Definitely valid
  • Completely different details? → Invalid (leave separate)

If valid duplicate → Decommission one record. Escalate if unsure.

Input data: Customer identity details, duplicate indicators
Processing responsibility: Staff (duplicate assessment)
Decision: Valid vs. invalid duplicate?
Output: Customer records corrected or confirmed separate
```

Q7 - Recalculate Liabilities IMPROVED

Select years within retrospective period (current + 4 prior only):

  1. Navigate to Liabilities node
  2. Click Recalculate Liability button
  3. Select ONLY years in retrospective period
  4. DO NOT select years outside retrospective period
  5. Click OK to process
Years outside retrospective period cannot be recalculated. Verify year selection before confirming.
Input data: Liability data, available years
Processing responsibility: Staff (year selection)
Decision: Which years to recalculate (retrospective only)?
Output: Liabilities recalculated for selected years

Completion Checklist (ALL required)

  • Address determined and documented
  • PPR verified and applied (or not, with reasons)
  • Duplicates resolved (if any)
  • Liabilities recalculated for all years
  • Assessment ready to generate
  • Deceased estate checked
  • Comments and notes recorded
  • QA/FTL comments completed
```
All checks complete? YES → Mark item complete / NO → Continue to Q9/Q10
Input data: FTL processing status (9-point checklist)
Processing responsibility: Staff (completion verification)
Decision: All requirements met? Ready to close?
Output: FTL action item marked complete
```

Q9 - Generate Assessment IMPROVED

Assessment Timing Rules

  • Current Year FTL: Assessment generated in normal cycle
  • Prior Year FTL: Assessment generated once cycle complete
  • Multiple Years: Separate assessment for each year
```

Generation Steps

  1. Verify liability recalculation complete
  2. Confirm assessment address correct
  3. Click "Generate Assessment" button
  4. Select year(s) requiring assessment
  5. Verify assessment generated
  6. Generate Late Issue Letter if prior year
Input data: Recalculated liabilities, current/prior year status
Processing responsibility: Staff (assessment generation)
Decision: Current year OR prior year? Which years?
Output: Assessment generated and issued
```

Q10 - Handle Deceased Estate (If Applicable) IMPROVED

Deceased Estate Detection

  • Check if customer is deceased (marked in e-Sys)
  • Confirm date of death (DLS or Probate Register)
```

Form 13 vs. Form 14 Decision

Scenario Form Type Action
Death < 1 year ago Form 13 (Intestacy) Create Form 13 action item
Death > 1 year ago Form 14 (Beneficiary) Create Form 14 action item

Apply deceased non-issue indicator to prevent assessment generation until estate resolved.

Input data: Deceased indicators, date of death
Processing responsibility: Staff (estate handling)
Decision: Deceased YES/NO? Form 13 or 14?
Output: Form 13/14 action item created or FTL continues
```

Retrospective Period Definition

Retrospective Period = Current calendar year + 4 prior years = 5 years total

```

Example (Year 2024)

  • Current: 2024
  • Prior 1: 2023
  • Prior 2: 2022
  • Prior 3: 2021
  • Prior 4: 2020
  • OUTSIDE PERIOD: 2019 and earlier (cannot process in FTL)

Rule: Only liabilities within 5-year period can be recalculated. Years outside cannot be amended via FTL.

```

Data Source Hierarchy - Q4 Address Determination

When multiple systems have conflicting information, use this hierarchy (most authoritative first):

```
Rank System Data Element Authority
1 RTBA Occupancy, rental address High (most recent if renting)
2 ATO Rental Rental address High (tax context)
3 DLS Current address, deceased status High (Government issued)
4 COI Corporate ownership, address Medium (ASIC cross-reference)
5 NOA Customer-reported address Medium (customer source)
6 e-Sys Current assessment address Low (SRO internal, may have errors)

Rule: Use most recent address from highest-ranked available source. Always document which source was used.

```

Definitions & Acronyms

FTL - First Time Liability (customer's first assessment for a property)
Retrospective Period - Current year plus 4 prior years (5 years total)
PPR - Principal Place of Residence (exemption from land tax)
e-Sys - SRO's property and liability management system
COI - Compliance Operational Intelligence (corporate/ownership data)
DLS - Driver Licensing System
RTBA - Residential Tenancy Bond Authority
ATO - Australian Taxation Office
Landata - Land titles online search system
NOA - Notice of Acquisition
V/F - Volume/Folio (unique land title identifier)
LOAN Complex Tray - Specialist team for complex liability cases
