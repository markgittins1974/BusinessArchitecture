Q1 - Is the valuation data file eligible for processing?

  • Receive Work Item: Work item contains file reference and council identifier.
  • Verify Receipt: Confirm files arrived in Outlook Valuations Data folder with all expected attachments.
  • Classify Data Type: Check email metadata and file names to determine Supplementary or Revaluation.
  • Check Prerequisites: Confirm no active investigations block processing of council or properties.
  • Decision: Is file complete and eligible for processing?
Do NOT proceed if files are incomplete or investigation is active. Escalate and halt processing.
Procedure reference: File Receipt and Initial Validation
Procedure Detail: Eligibility Criteria▼

File Completeness Check

Mandatory files (all 4 required): Type 1 data, Type 2 data, Type 3 data, VGV reference spreadsheet.

Optional files (conditional): Large value properties Excel file (5 councils only), Alpine resort data files (2 councils only).

Data Type Classification

Supplementary Valuation: Quarterly receipt, file naming includes "SUPPS", smaller file sizes.

Revaluation: Annual receipt, file naming includes "REVAL" or "VALUATION", larger file sizes.

Prerequisite Verification

Search CM9 for active investigations on council. Check investigation start/end dates. If investigation end date is before file receipt date, file may proceed. If active investigation, escalate to Investigation team.

Source: File Receipt and Initial Validation
Input data: Work item, email with file attachments, investigation status
Processing responsibility: LRA staff (receipt verification), CM9 (investigation check)
Decision: Complete file set + no investigation = proceed. Incomplete OR investigation = halt
Output: Data type classified, work item status updated, files marked for processing

Q2 - Are all required files present and accessible?

  • Save Files: Move files from email to OneDrive (LGAT - Supps Formatting) in council/year/version folder.
  • Handle Automation Failures: If automation failed, manually save using "yyyy-mm-dd" folder naming convention.
  • Register Batch: Record batch reference, council, receipt date, file count in appropriate register (Supps or Reval).
  • Verify Storage: Confirm all files copied to destination and are accessible from working directory.
  • Decision: Are files saved and registered correctly?
System Note: Automation may fail; fallback to manual saving is expected. OneDrive folder structure is enforced by register system.

Q3 - Does data require preparation before formatting?

  • Assess Pathway: Determine if standard processing or special preparation is needed.
  • Standard Path: Most supplementary files require file amalgamation only. Proceed directly to Q4.
  • Large Value Properties Path: 5 councils (Melbourne, Bass Coast, Latrobe, Moonee Valley, Wellington) require field mapping if properties exceed 9-digit value limit.
  • Alpine Resort Path: 2 councils (Alpine Shire, Mansfield Shire) require data merge if resort files provided.
  • Return Date Check: Revaluation files must have position 280-288 = "010720XX" (1 July). Correct if different.
System Note: Only TextPad "Insert Files" method is approved for amalgamation. File count verification prevents file loss.
Procedure reference: Preparation Pathways and Methods
Procedure Detail: Preparation Sub-Processes▼

Standard Supplementary Path

File amalgamation using TextPad. Open blank Type 1 template. Select Edit > Insert Files. Choose all Type 1 received files. Repeat for Type 2 and Type 3. Verify file count matches received count. Save files.

Large Value Properties Preparation (5 councils)

Extract LVP data from Excel spreadsheet provided by VGV. Map property details (not values) into Type 1 file. Insert zero for all value fields in Type 1. Create separate Type 2 entries for SV/CIV/NAV values. Use previous year file as template. Post-load manual update required in e-Sys.

Alpine Resort Data Preparation (2 councils)

Merge resort data with council data using TextPad. Open blank Type 1 template from S:\FTP\Val_Returns\Textpad Templates. Insert Files > select council file + resort file. Save as merged file. Repeat for Type 2 and Type 3.

Return Date Validation (Revaluation Only)

Type 1 file positions 280-288 must be "010720XX" (1 July of relevant year). If incorrect, manually correct before formatting. This date defines assessment period.

Source: Preparation Pathways and Methods
Input data: Classified files from Q2, council identifier, data type
Processing responsibility: LRA staff (pathway selection, preparation), TextPad (file insertion)
Decision: Standard = no prep | LVP = field mapping | Resorts = data merge | Return date wrong = correct
Output: Prepared files ready for e-Sys formatting tool, or files pass through unchanged

Q4 - Do files pass e-Sys validation?

  • Launch Tool: Open e-Sys Formatting Municipal Revaluation Files tool from Tools menu.
  • Enter Parameters: Municipality code, File type (Supps/Reval), Valuation Year (assessment year, not valuation year).
  • Validate Sequentially: Type 1 → Type 2 → Type 3 (one at a time, in order). Resolve exceptions after each.
  • Generate Type 4: Click "Create Type 4 Totals" to auto-generate summary file from validated Types 1, 2, 3.
  • Save All Files: Save outputs to S:\FTP\Val_Returns\[YEAR]\[COUNCIL]\formatted\ directory.
Do NOT proceed if exceptions remain. Exception resolution is MANDATORY before file acceptance. Do not use "Select All" button (includes years outside retrospective period).
Procedure reference: E-Sys Formatting Tool and Exception Resolution
Procedure Detail: Exception Resolution by Type▼

Type 1 Validation - Duplicate MPN Check

Rule: No duplicate Municipal Property Numbers allowed. Resolution: Investigate source data. Remove duplicate or correct MPN. Re-validate. Escalate if: >5 duplicates found (data quality issue).

Type 2 Validation - Excess Lines and SPI Correction

Excess Lines Rule: Max 50 lines per Municipal Number. If error: Click Yes, tool highlights MPN, right-click "Remove excess records", re-validate. Repeat until no excess found. Escalate if: >10 MUNs affected.

SPI Correction Rule: Field 373 must be valid SPI or "0". If exception: Highlight exception, right-click "Navigate to error", correct SPI value. For complex reference strings (multiple TPs), manual SPI creation required. Re-validate until all SPI exceptions show "SPI 0" message.

Type 3 Validation - Pre-Fill Verification

Rule: Tool pre-fills missing names with "unknown" and addresses with "not provided". Verify pre-fills are acceptable. If incorrect, update manually and re-validate.

Type 4 Generation

No validation needed. Click "Create Type 4 Totals". Tool auto-generates from Types 1, 2, 3. File ready for next step.

Source: E-Sys Formatting Tool Operation and Exception Resolution
Input data: Prepared files from Q3, municipality code, file type, assessment year
Processing responsibility: LRA staff (tool operation, exception resolution), e-Sys tool (validation, pre-filling)
Decision: All exceptions resolved = proceed | Exceptions remain = resolve or escalate
Output: Type 1,2,3,4 validated and saved. Files ready for Q5. Exception report generated.

Q5 - Are files documented and ready for testing?

  • Verify File Locations: Confirm all 4 file types exist in S:\FTP\Val_Returns\[YEAR]\[COUNCIL]\formatted\ directory.
  • Check File Integrity: All files must be non-empty (>0 bytes). Verify using file properties.
  • Document Work Item: Record batch reference, council, receipt date, data type, file count, special handling notes.
  • Quality Checklist: Confirm: All 4 files present, exception count ≤10, files in correct location, documentation complete.
  • Decision: Are files approved for testing?

Q6 - Are mandatory sequencing requirements satisfied?

  • Sequencing Verification: Confirm Q1→Q2→Q3→Q4→Q5 completed in order. No backward loops permitted.
  • File Location Check: Files must be in S:\FTP\Val_Returns\[YEAR]\[COUNCIL]\formatted\ for Testing team access.
  • Work Item Status: Status must be updated to reflect current stage. No stalled work items.
  • Escalation Review: If escalations occurred, confirm they were resolved and approved before proceeding.
  • Decision: Can files proceed to Testing Procedure?
If mandatory prerequisites not satisfied: Do NOT advance. Resolve blockers before proceeding to Testing.

Procedure Detail: Sequencing Rules▼

Mandatory Execution Order

Q1 Eligibility → Q2 Registration → Q3 Preparation → Q4 Validation → Q5 Documentation → Q6 Sequencing Gate → Q7 Handoff

Blocking Dependencies

  • Q4 blocks Q5: If exceptions unresolved, cannot advance to documentation
  • Q5 blocks Q6: If files not verified, cannot approve for testing
  • Q6 blocks Q7: If sequencing not verified, cannot handoff to Testing

No Backward Loops Rule

Once Q5 complete (files documented), cannot return to Q3 (preparation). Cannot re-run formatting after Q4 complete unless critical issues discovered. Rework requires new batch submission.

Critical Gaps & Recommendations

GAP 1 (HIGH): Return date determination unclear. LGA example shows "29/01/xx" but rationale undefined. Fix: Define rule: use decommission discovery date OR assessment year end (01 July). Document choice in comments.
GAP 2 (HIGH): "Select All" button trap in e-Sys. Users may accidentally include years outside retrospective period (current + 4 years). Fix: Mandatory warning: Never use "Select All". Manually select years only within retrospective period.
GAP 3 (MEDIUM): SPI creation algorithm not documented. For complex reference strings (multiple TPs), rule is "cannot auto-generate, manual creation required" but process is undefined. Fix: Create SPI format rules or reference external Property Reference Code documentation.
GAP 4 (MEDIUM): Exception handling lacks fallback rules. If e-Sys tool cannot resolve exception, no documented escalation path. Fix: Define: escalate to Grade 3 if >10 unresolved exceptions remain after 2 iterations.
GAP 5 (MEDIUM): File location terminology inconsistent. References to S: drive, OneDrive, H: drive locations scattered. Fix: Create single "System Architecture" section defining all storage locations and usage.
GAP 6 (MEDIUM): Work item lifecycle undefined. No documented states: received → validating → exception resolution → approved → handed off. Fix: Create status update rules at each Q-gate.
GAP 7 (LOW): No defined SLA for each Q-gate. Formatting duration varies (1-3 days) but no formal time targets. Fix: Add target durations: Q1 (1 day), Q2 (1 day), Q3 (1 day if LVP/resorts, <1 day standard), Q4 (1-2 days exception resolution), Q5-Q7 (<1 day).
