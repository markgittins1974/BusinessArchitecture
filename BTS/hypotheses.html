Case Management Solution – How Might We Hypotheses

```

Objective: Translate operational pain points from the SRO investigations workflow into testable "How Might We" statements that a case management system could address. Each HMW focuses on friction points that recur across revenue lines or are revenue-line-specific bottlenecks.

📊 Measurable Indicators: HMW statements marked with 📊 have clear, quantifiable success metrics. Unmarked statements are qualitative or harder to measure but still important.

VALUE STAGE 1: INTAKE & VALIDATION

Pain: Inconsistent PIC Quality & Multiple Intake Channels
Root Cause
PICs come from multiple sources (email, systems, third parties, referrals) with no standardisation. Data entry happens 3+ times across ESYS, CM9, email.
"PICs come from everywhere — no consistency." / "We enter the same info in three places."
How Might We Statements
  • How might we create a single, standardised intake form that captures all required data once and syncs to all backend systems (ESYS, CM9, COI)? [Metric: % of cases with duplicate data entry reduced to 0; data sync latency <5min]
  • How might we implement intake rules that route different PIC types (email, referral, system alert, third-party) to standardised workflows automatically?
  • How might we validate PIC completeness at intake, flagging missing information and prompting investigators to fill gaps before case creation? [Metric: % of case records flagged for incomplete data at intake; % rework due to incomplete PIC data]
  • How might we create a PIC template library (by revenue line) that auto-populates common fields based on trigger type?
All Revenue Lines

VALUE STAGE 2: TRIAGE & PRIORITISATION

Pain: Data Integration Fragmentation & Manual Reconciliation
Root Cause – All Revenue Lines
LT: Valuation Services data conflicts with council data. PTX: STP/VWA reconciliation gaps. FHOG: Utilities data delays. Duties: Conveyancer uploads inconsistent.
"We spend half our time Googling." / "Grouping is a nightmare — there's no single source of truth."
How Might We Statements
  • How might we integrate council, Valuation Services, STP, VWA, and utilities data sources into a single case dashboard that automatically reconciles conflicts and flags discrepancies? [Metric: % of data integration enabled by value stage 2; time saved on manual reconciliation; % of discrepancies flagged vs missed]
  • How might we create a data quality dashboard showing coverage, staleness, and completeness of each external data source for a given case?
  • How might we implement an automated reconciliation workflow that identifies and resolves VWA/STP wage discrepancies without manual intervention? [Metric: % of wage discrepancies auto-resolved; time saved per PTX case during triage]
  • How might we create a council data sync that automatically updates case records when new council rates, valuation, or occupancy data becomes available? [Metric: latency of council data updates <24hrs; % of LT cases with auto-updated council data]
LT PTX FHOG Duties
Pain: No Consistent Risk Framework & Decision Criteria
Root Cause
Risk assessment is subjective. No standardised risk framework. Early closure criteria missing. Cases progress unnecessarily.
"Risk is subjective — everyone does it differently." / "We don't close things early enough — they drag on."
How Might We Statements
  • How might we embed a configurable risk scoring model (by revenue line) that automatically calculates case risk and priority based on trigger type, entity characteristics, and historical patterns? [Metric: % of cases risk-scored automatically; consistency of risk scores across investigators; time saved on manual risk assessment]
  • How might we create a set of early-closure decision rules that flag low-value cases for immediate redirect/close without full investigation? [Metric: % of cases closed early; avg cycle time reduction; cost savings per case avoided]
  • How might we surface risk factors and decision rationale transparently so investigators understand why cases are scored/prioritised as they are?
  • How might we track risk framework consistency and alert managers when risk assessment outliers appear (e.g., an investigator scores 80% of cases as high-risk)? [Metric: % of investigators with risk score variance >2σ; number of consistency alerts triggered]
All Revenue Lines

VALUE STAGE 3: INVESTIGATION PLANNING

Pain: No Structured Planning & Manual Approvals
Root Cause
Plans live in heads, not documented. Approvals entirely email-based. No auditability. No handover capability. Revenue lines have different scope considerations.
"Plans live in our heads — nothing is written down." / "Approvals take forever — everything is via email."
How Might We Statements
  • How might we create investigation plan templates (by revenue line) that guide investigators through scope definition, evidence identification, and task breakdown?
  • How might we implement a workflow approval system that routes plans to the right approver (Team Leader, Legal, Executive) based on case complexity and power usage? [Metric: % of plans approved via system; avg approval time; bottlenecks identified]
  • How might we track plan versions, approvals, and changes so that any modifications create an audit trail? [Metric: % of plans with audit trail; number of plan changes tracked; time to retrieve approval history]
  • How might we auto-generate task lists from investigation plans and assign them with SLAs to prevent scope creep and ensure progress visibility? [Metric: % of scope creep incidents; task SLA compliance; avg cycle time deviation from plan]
All Revenue Lines

VALUE STAGE 4: EVIDENCE & ANALYSIS

Pain: Fragmented Evidence Tracking & Document Chaos
Root Cause – Revenue Line Specific
LT: Duplicate evidence requests (council data not integrated). PTX: 20+ payroll formats, manual standardization. FHOG: Document fraud risk, extensive verification. Duties: 200-page contracts with no indexing.
"We chase the same documents multiple times." / "Employers give us payroll data in 20 different formats." / "We get 200-page contracts with no index."
How Might We Statements
  • How might we create a centralised evidence registry that tracks what evidence has been collected, when, from which source, and whether it's been verified/analysed? [Metric: % of evidence tracked in registry; duplicate evidence requests eliminated; time to locate evidence]
  • How might we implement evidence requirement checklists (by revenue line) that auto-populate based on investigation plan and flag when evidence is still outstanding? [Metric: % of evidence gaps identified; avg time to identify missing evidence; cases with 100% evidence completeness]
  • How might we auto-normalise payroll data formats so that employers can upload in any format and the system converts to a standard schema? [Metric: % of payroll data auto-normalised; time saved on PTX manual standardization; format variance reduced]
  • How might we implement intelligent document parsing for large contracts (Duties) that auto-indexes sections and extracts key clauses (consideration, surcharge applicability, etc.)? [Metric: % of key clauses auto-extracted; time saved on Duties contract review; accuracy of extraction]
  • How might we create a document verification workflow (for FHOG) that flags suspicious documents and surfaces anomalies (e.g., utilities data showing zero usage)? [Metric: % of document fraud detected; false positive rate; time saved on verification]
LT PTX FHOG Duties
Pain: Inconsistent Evidence Analysis & No Standardised Criteria
Root Cause
Everyone interprets evidence differently. No standardised assessment criteria. Manual interviews, no structured templates. Inconsistent outcomes.
"Everyone interprets evidence differently." / "We don't have a standard way to record interviews."
How Might We Statements
  • How might we create decision trees / assessment frameworks (by revenue line) that guide investigators through evidence evaluation and help them reach consistent conclusions? [Metric: % of cases using assessment framework; consistency of preliminary view across multiple investigators; objection override rate]
  • How might we implement interview templates (with guided questions, decision logic) that ensure interviews are recorded consistently and all material facts are captured? [Metric: % of interviews using template; completeness of interview records; time saved on interview documentation]
  • How might we track preliminary views and their supporting evidence so that multiple investigators assessing the same facts reach similar conclusions? [Metric: consistency score for preliminary views; % agreement between investigators on similar cases; variance analysis]
  • How might we flag inconsistencies when evidence contradicts preliminary view assumptions, prompting re-analysis before findings are drafted? [Metric: % of inconsistencies flagged; re-analysis triggered; findings accuracy improved]

Root Cause – Revenue Line Specific
LT: 4+ notice variants (Assessment, VRLT, Land Use, Trust Surcharge). PTX: Grouping notices high dispute. FHOG: Reversal notices emotionally sensitive. Duties: Foreign surcharge contested.
"Templates are inconsistent — everyone writes them differently." / "LT notices are the most complex — too many variations."
How Might We Statements
  • How might we create smart notice templates (by revenue line and notice type) that auto-populate key findings and calculations from the case record? [Metric: % of notices auto-generated; time saved drafting notices; consistency of notice content]
  • How might we implement dynamic notices that adjust language based on case facts (e.g., FHOG reversals include hardship language, foreign surcharge notices include exemption summary)?
  • How might we build a legal review workflow that validates notices for completeness, consistency, and legal compliance before issuance? [Metric: % of notices legally reviewed; % of legal errors caught pre-issuance; review time]
  • How might we create a precedent library of historical notices (by revenue line) that investigators can use as reference when drafting new notices?
LT PTX FHOG Duties
Pain: No Structured Objection Handling & Resolution Delays
Root Cause – Revenue Line Specific
LT: Valuation Service dependency creates delays. PTX: Objections trigger full re-analysis. FHOG: Emotional appeals require careful handling. Duties: Technical complexity slows resolution.
"Objections go back and forth forever." / "Objections turn into mini-audits — we basically redo the whole case."
How Might We Statements
  • How might we implement a structured objection workflow that categorises objections (e.g., factual dispute, legal dispute, new evidence) and routes them to the right handler with SLA? [Metric: % of objections categorised; routing accuracy; SLA compliance; avg objection resolution time]
  • How might we create templates for common objections (by revenue line) so that investigators can respond consistently and quickly? [Metric: % of objections responded via template; time saved per objection response; consistency of responses]
  • How might we track objection dependencies (e.g., Valuation Service involvement) and monitor external delays so they don't fall through the cracks? [Metric: % of dependency-blocked objections tracked; number of items falling through; time to resolution increase]
  • How might we surface objection re-analysis requirements early so PTX cases aren't re-audited from scratch, but only affected sections are re-analysed? [Metric: % of objections requiring full re-analysis vs partial; time saved on PTX re-analysis; rework cost reduction]
  • How might we implement escalation rules for high-emotion cases (FHOG) that route to senior investigators or add communication support?

  • How might we tag and search insights by revenue line, actor, issue type so that new investigators can learn from past cases? [Metric: # of insight searches; insights applied to new cases; time to relevant past case discovery]

Summary: Case Management Solution Hypothesis Map

Core Hypothesis: A centralized case management system that integrates data from fragmented sources, enforces standardised workflows, and embeds domain logic (risk models, assessment frameworks, decision rules) will reduce investigation cycle time by 30-40% and improve consistency across revenue lines.

Critical Success Factors:

  • Data Integration: Single source of truth for all case data (PIC, council, Valuation Services, STP, VWA, utilities, contracts). Reconciliation and conflict detection built in.
  • Workflow Standardisation: Enforced intake, triage, planning, evidence, findings, objection handling. Revenue-line-specific templates and rules.
  • Decision Support: Risk models, assessment frameworks, objection decision trees embedded in workflows. Transparent, consistent, auditable.
  • Evidence Management: Centralised registry, auto-normalisation, intelligent parsing, fraud detection. Prevents duplicate requests and rework.
  • Learning Loop: Structured capture, aggregation, and sharing of insights. Feedback to improve risk models, templates, and rules.

Revenue Line Specific Opportunities:

  • Land Tax: Council data integration, valuation conflict resolution, trust structure analysis automation, exemption eligibility rules.
  • Payroll Tax: STP/VWA reconciliation, payroll format normalisation, grouping relationship mapping, contractor classification decision tree.
  • FHOG: Utilities data integration, occupancy verification workflow, document fraud detection, sensitive communication escalation.
  • Duties: Valuation services integration, foreign surcharge logic, landholder threshold automation, large contract parsing.
