Q1 - Does the application identify the correct property and vendor?

  • Launch PIA work item. Check Details tab for additional comments/title references.
  • Property search: Search using all references (Volume/Folio, Lot/Plan, Address, Vendor). If not found in e-Sys, search COI (may have previous references). Check New Plans folder for plans received but not yet processed.
  • Property clarification (p.16): If property details don't match e-Sys/COI:
    • First two references match property and owner: issue certificate (only first two appear on cert).
    • Only one of first two matches: contact applicant for clarification.
    • None match: contact applicant for clarification.
  • Non-rateable property: In Details tab enter comment that property is not rated by council. Select relevant exemption from dropdown.
  • Associate property: If correct property is not already associated to PIA, return to property address under PIA node, select edit, associate the correct property.
  • Vendor validation: Check if the owner of the property is the same as the vendor on the PIA. If no, initiate Vendor Clarification process.
  • Seven acceptable vendor types: Current registered proprietor on title, Purchaser (with current proprietor name on application), Mortgagee in possession (major bank), Power of Attorney (evidence required), Executor of deceased estate of current registered proprietor, On-seller, Company Director / Crown Legal Interests.
  • If vendor is a lessee (not registered proprietor) of a business or property: redirect PIA to Cancellation Tray.
  • Vendor amendment rules (p.17): First/last name only = update and issue. Surname not provided = update once confirmed. Company name incomplete = delete "Not provided" and issue. Purchaser/vendor reversed = write comment and issue. Vendor not acceptable = confirm with applicant, amend PIA details.
Certificate cannot be issued if vendor is listed as a tenant/lessee. Redirect to PIA Cancellation Tray.
Procedure reference: Application (p.4); Vendor Clarification (p.9); Property Clarification (p.16); Amending application (p.17)
Procedure Detail: Application and Vendor Clarification ▼

Application Flow (p.4)

Launch PIA work item. Check Details tab for any additional comments or title references. Search for property using all available references: Volume/Folio, Lot/Plan, Address, Vendor. If property cannot be located, search references in COI (COI may have previous property references). Search plan number in New Plans folder (plan may have been received but not processed).

Subdivision Check

If property has been created in the current year or is part of a planned subdivision, go to Pro Rata process (Q3). If not a subdivision, continue with standard processing.

Copy of Plan (p.8)

If a full copy of plan is required: check if property is in e-Sys (property address, U200 screen, description/title tab, right-click plan number, view electronic plan). If temporary folder opens, search New Plans folder. If plan cannot be located, check Dye and Durham website for other applications with same plan number in current year. If plan not in attached documents, check if agency application (send template email) or eBus applicant (phone directly). Place comment on PIA noting request date and user ID.

Vendor Clarification (p.9)

If vendor on e-Sys does not match vendor listed in PIA: For companies, check if vendor is a company on e-Sys/COI and application has vendor as a person. Use ASIC to check directors. If listed as director, place note in details confirming and process PIA.

For eBus applications: contact applicant directly to clarify. After multiple attempts, use e-Bus applicant template email. While waiting, note "Vendor Clarification requested" in PIA details tab and place in reserve.

For Agency applications: check if vendor provided is valid. Examples of invalid: only first or last name, company name only, TBC/Unknown. If agency entered vendor in acceptable format, process. If not, send email to agency using Vendor Clarification template with application number and User ID.

Acceptable Vendors

  • Current Registered Proprietor on title
  • Purchaser (must have current registered proprietor name on application)
  • Mortgagee in possession (e.g. major bank - Westpac, ANZ, Commonwealth etc)
  • Power of Attorney (evidence required)
  • Executor of deceased estate of current registered proprietor on title
  • On-seller
  • Company Director and Crown Legal Interests

If applicant is other than current registered proprietor, comment in application: e.g. "Vendor is current Proprietor"

Vendor Amendment Rules (p.17)

  • Only first or last name of individual/s: once confirmed, update and issue (some people only have a first name)
  • Not provided with Surname: once confirmed, update and issue
  • Not provided with First name: once confirmed, update and issue
  • Not provided with Company name: delete "Not provided" and issue
  • Company full name not provided: if enough matches title, issue
  • Not all names on title provided: issue (at least one correct name must be provided)
  • Purchaser and Vendor names reversed: write comment "purchaser and vendor reversed" and issue
  • Vendor does not match and is not acceptable: confirm with applicant, amend PIA details
  • Lessee (not registered proprietor): redirect to PIA Cancellation Tray
Source: Application (p.4); Copy of Plan (p.8); Vendor Clarification (p.9); Property/Vendor Clarification (p.16-17)
Input data:
  • PIA application details (property references, vendor name, purchaser name)
  • Property search references: Volume/Folio, Lot/Plan, Address, Vendor
  • e-Sys property records
  • COI title information (may have previous references)
  • New Plans folder (H:\LandTax\...\Valuations Team\Plans)
  • ASIC (company director verification)
  • Application source: eBus or Agency (Landata, Dye and Durham, Infotrack)
Processing responsibility:
  • CSO - property search, vendor validation, clarification requests
  • Senior CSO - escalation for complex clarifications
  • e-Sys - PIA work item, property association
  • COI - title verification
  • ASIC - company director checks
Decision: Property and vendor validation:
  • Property found and vendor matches: proceed to Q2
  • Property found, vendor mismatch: Vendor Clarification (7 acceptable types)
  • Property not found: Property Clarification (3-tier reference matching)
  • Vendor is lessee/tenant: redirect to PIA Cancellation Tray
  • Subdivision detected: redirect to Q3 (Pro Rata)
Output:
  • Property validated and associated to PIA
  • Vendor confirmed or clarification requested (PIA placed in reserve)
  • Or PIA cancelled (invalid vendor)

Q2 - Can the certificate be issued via automation?

  • e-Sys has automation rules. If all information matches and no warnings, PIA auto-issues within 24 hours.
  • If warnings or errors appear, manual intervention is required. There are 12 distinct warning types, each with its own resolution path:
  • 1. Property Warning (multiple refs): Check all references for related properties. Process as normal with comment "Verified/Resolved."
  • 2. Property Warning (not current year): Pro rata required on parent property. See Q3.
  • 3. Settlement Date Warning: If settlement is more than 3 months away, confirm with applicant. If correct, leave PIA in tray — auto-issues in January.
  • 4. Comment Warning: Review and record. See Senior CSO.
  • 5. Recalc Warning: Resubmission trigger. See Senior CSO.
  • 6. Non-Issue Warning: Base indicator set for liable party. Check non-issue code and IOs. Can override if non-issue doesn't affect certificate.
  • 7. Special Customer Warning (current year): Special customer impact detected. See Special Customer process.
  • 8. Special Customer Warning (previous years): Override warning and issue certificate.
  • 9. Liability Warning (CIPT/LTX): Certificate impacted by CIPT or land tax not yet assessed. Override, review tolerance. See Senior CSO.
  • 10. Valuation Warning: Missing valuations. Check if property is exempt via exemptions node under property and customer.
  • 11. Duplicate Warning: Duplicate listing detected. See Senior CSO.
  • 12. Customer Warning: Auto-certificate generated previously. Check if single customer. If not, see Senior CSO, redirect to Land Taxpayer PIA Tray.
  • 13. Liability Warning (unable to generate): Unexpected order of events. Customer payment process required.
Do not issue where unresolved non-issue indicators or blocking errors apply. Escalate to Senior CSO.
Procedure reference: PIA Warnings (p.3)
Procedure Detail: PIA Warning Matrix (12 types) ▼

How Automation Works

There are rules set up in e-Sys that allow certificates to go through automation and issue by e-Sys. There are times where the application will require manual intervention. Warnings will need to be resolved before the user can issue the certificate.

Warning 1: Property Warning (multiple references)

Property has multiple references and requires manual matching. Check if all references are related to the same property. Search for additional properties if required. If there are more references to the property, enter them and add relevant comment. Process as normal - include text "Verified" or "Resolved" in comment.

Warning 2: Property Warning (not in current calendar year)

Property is not in the current assessment year. Pro rata is required on parent property. See Pro Rata process (Q3).

Warning 3: Settlement Date Warning

Settlement/certificate date is in a future year. If settlement date is more than 3 months away, confirm with applicant there is no data entry error. If correct, leave PIA in the tray - it will issue automatically on first business day in January.

Warning 4: Comment Warning

Reading/recalculation trigger detected. Review and record comment. See Senior CSO.

Warning 5: Recalc Warning

Resubmission has been triggered. See Senior CSO.

Warning 6: Non-Issue Warning

A base indicator is set for the Property Clearance Certificate on a current assessment year. Check the non-issue code under the customer and any IOs (Investigation Orders) that the customer file may have. Override the warning and issue certificate if the non-issue does not impact the certificate.

Warning 7 and 8: Special Customer Warning

Warning 7: Special customer impact on current year detected. See Special Customer process. Warning 8: Special customer impact on previous years only. Override warning and issue certificate.

Warning 9: Liability Warning

Certificate impacted by CIPT (Commercial and Industrial Property Tax) or land tax not yet assessed. Override warning, review tolerance/relevance. See Senior CSO if unsure.

Warning 10: Valuation Warning

Missing valuations detected. Check if the property is exempted by looking at the exemptions node under the property and customer. If exempt, property should not have a valuation - override.

Warning 11: Duplicate Warning

Duplicate listing of an e-Sys property detected. See Senior CSO.

Warning 12: Customer Warning

Auto-certificate has been generated previously for this customer. Check if the customer is a single customer. If not (e.g. merged/duplicate), see Senior CSO. May need to redirect PIA to Land Taxpayer PIA Tray.

Warning 13: Liability Warning (unable to generate)

Unable to generate due to unexpected order of events. Customer payment process required. See Senior CSO.

Source: PIA Warnings flowchart (p.3)
Input data:
  • PIA warnings and errors (12 distinct types)
  • Non-issue indicators and codes
  • Investigation Orders (IOs)
  • Settlement/certificate date
  • Special customer flags
  • CIPT/LTX assessment status
  • Valuation records
  • Duplicate property flags
  • Prior certificate issuance history
Processing responsibility:
  • e-Sys - automation rules, warning generation
  • CSO - warning resolution for straightforward cases
  • Senior CSO - escalation for warnings 4, 5, 9, 11, 12, 13
Decision: Warning resolution (per warning type):
  • No warnings: auto-issue within 24 hours
  • Resolvable warning: CSO resolves, override, continue
  • Senior CSO required: escalate (warnings 4, 5, 9, 11, 12, 13)
  • Blocking error: manual processing or cancellation
  • Future settlement: hold in tray for January auto-issue
Output:
  • Warnings resolved - proceed to certificate generation
  • Or PIA held, escalated, or redirected
High-cardinality branching: 12+ warning types each with distinct resolution paths. Some require override, some require Senior CSO, some redirect to other trays. This is the most complex decision matrix in the procedure.

Q3 - Is pro rata calculation required for the property?

  • Determine if property is part of a subdivision or planned development (created in current year).
  • If property is in e-Sys: click Related Properties to identify parent/grandparent. If not in e-Sys: search New Plans folder or request copy from applicant.
  • Load parent property/ies and vendor to tree. Check if vendor has prior year liabilities.
  • If property start date is prior to 31 December of the year before: property cannot be used. Go back to grandparent properties.
  • Area vs Unit Entitlement: Must use area if available. Search for area or unit entitlement via New Plans folder.
  • Pro rata steps: Under PIA node, select property address, associate parent property/ies. Select Land ID. Click Edit and Pro Rata. Enter area of child. Select parent properties via tick boxes.
  • 60/40 Rule (p.6): If subdivision has roads or common property, apply Parent Area Discount (tick box). Ensure parent area discount box is unchecked for the child. This ensures land tax on roads/common property is collected when all child properties are sold.
  • When 60/40 does NOT apply: Large single lots without roads/common property (e.g. single stage lot, broad-acre land split).
  • Unit Entitlement (p.7): If no child area/unit entitlement, issue certificate on parent property with comment "This certificate has been issued on the parent property." If plan has mixed UE and square meters, issue without pro rata with same parent property comment.
Procedure reference: Pro rata - Area of Unit Entitlement (p.5); Different types of Pro rata / 60/40 Rule (p.6-7); Copy of Plan (p.8)
Procedure Detail: Pro Rata and 60/40 Rule ▼

Property Determination

Has the property been created in the current year or is it part of a planned subdivision? If property is created in current year: click Related Properties to identify parent/grandparent property/ies. If property is not located in e-Sys: search New Plans folder or request a copy of the plan from the applicant.

Parent Property Traversal

Search for parent/grandparent properties using references provided on the new plan. Load parent property/ies and vendor to tree. Does the vendor have prior year liabilities? If yes, refer to Senior CSO for Manual Certificate. If no, check: is property start date prior to 31 December of the year before? If yes, property cannot be used - go back to grandparent property/ies. If no (start date is on or after 31 Dec), proceed with pro rata.

Area vs Unit Entitlement

Is there an area or unit entitlement on the property to use for pro rata? Must use area if available. Search for area or unit entitlement via New Plans folder. If area/UE cannot be found for all properties, click on Liability node and enter comment "Certificate has been issued on the parent property." Issue certificate on parent property with comment "This certificate has been issued on the parent property."

Pro Rata Steps in e-Sys

Under PIA node, select property address and associate the parent property/ies. Select Land ID (or 1 of if multiple properties). Click Edit and Pro Rata. Has land area been supplied? If no, check other sources for area (electronic plan, other certs issued under parent property). If yes, enter area of child property. Select area for pro rata basis (must use area if available). Select parent property/ies by checking tick boxes. Enter area of child.

60/40 Rule

If the new plan of subdivision has roads or common property, apply the Parent Area Discount (defaulted with a tick box). This is the 60/40 rule - a calculation that takes into consideration roads or common property, so that when all child properties are sold, the land tax applicable to roads and common property is collected. Make sure parent area discount box is unchecked for the child property. Click OK and continue online PIA process to issue certificate.

Note: Not all pro rata scenarios require the 60/40 rule. Large single lots (e.g. if applicant has applied for a single lot of a large area), broad-acre land splits without roads or common property do NOT require 60/40.

Unit Entitlement

If you do not have a child area/unit entitlement, issue the certificate on the parent property/ies with comment "This certificate has been issued on the parent property." If the plan has some child properties with unit entitlements and other child properties with square meters, issue the certificate without a pro rata with the parent property comment. New Plans folder will contain an Owners Corporation report if UE are required for pro rata.

Manual PIA Pro Rata (p.21)

For inactive parent properties with net liabilities: requires LTX-A_SPWF e-Sys profile. Each pro rata calculation must be done separately. Can associate multiple parent properties. Certificate displays child liability and child's portion of parent/grandparent liabilities separately. Liability fields can be overwritten.

Scenario 1: If parent properties have different liability dates, pro rata must be done for each individually. Scenario 2: If multiple parent properties share the same year liability, a single pro rata can cover them, but additional liabilities unique to one parent require a separate pro rata using the manual calculator spreadsheet.

Source: Pro rata (p.5-7); Copy of Plan (p.8); Manual PIA (p.21)
Input data:
  • Property creation date (current year or part of planned subdivision)
  • Parent/grandparent property references
  • Area or Unit Entitlement data from plans
  • New Plans folder (electronic plans, Owners Corporation reports)
  • Vendor prior year liabilities on parent properties
  • Roads/common property in subdivision (triggers 60/40)
Processing responsibility:
  • CSO - pro rata calculation, plan sourcing
  • Senior CSO - manual certificate for prior year liabilities
  • e-Sys - PIA node, property association, pro rata calculation
  • Dye and Durham - plan sourcing for agency applications
Decision: Pro rata type:
  • No subdivision: skip to Q4
  • Subdivision with area data: area-based pro rata
  • Subdivision with UE data: unit entitlement pro rata
  • Subdivision with roads/common property: apply 60/40 rule
  • No child area/UE available: issue on parent with comment
  • Mixed UE and square meters: issue without pro rata with parent comment
Output:
  • Pro rata correctly applied in e-Sys
  • Or certificate issued on parent property with comment
  • Or manual certificate required (Senior CSO)

Q4 - Is ownership confirmed for the relevant assessment period?

  • Owner Unknown: Does vendor on application match e-Sys? If ownership is recorded as "Owner Unknown," certificate cannot be issued. Check if authorised for LO updates. If yes, process from LO PIA tray. If no, leave note "Owner unknown, redirected to LO tray" and redirect.
  • COI Check: Check COI using Vol/Fol or other title details. Does vendor match registered proprietor in COI? If no, initiate Vendor Clarification (Q1). If yes, check date of possession.
  • Date of possession: Is vendor the registered proprietor on or before 31 December of current assessment year? If on or after: process PCC as is. If before: check if date impacts prior assessment years. If impacts prior years, check if authorised for LO updates.
  • Deceased Estate (p.11): Check COI for "transmission application" dealing type. If yes:
    • Date of death less than 3 years: load deceased customer, enter DOD, add comment with source (Supreme Court, probate websites, SAI Global, LPR etc). Place comment in PIA: "Trust not created. Vendor is exec." Process as normal.
    • Date of death more than 3 years: check if estate is already a Trust customer type. If yes, redirect to Estate tray. If no, redirect to Estate tray for escalation to ERC with note "DOD as per Probate for PIA Number *****."
  • Determine executors: Check COI. Search Supreme Court (supremecourt.vic.gov.au), probate records (online.justice.vic.gov.au), Herald Sun tributes. If date not found/known, request clarification via agency or eBus applicant.
Certificate cannot be issued as "owner unknown." Redirect to LO tray.
Procedure reference: Change of Ownership Required (p.10); Deceased Estate (p.11)

Change of Ownership Required (p.10)

These are complex and can take more than the 6-minute certificate time stamp. Processing and management of the Land Ownership tray falls with the PCC BTO.

Owner Unknown Path

Does vendor on the application match e-Sys? If no, is ownership in Owner Unknown? If yes: note that certificate cannot be issued as "owner unknown." Check if you are authorised to complete LO updates. If yes, continue to "Processing from the LO tray." If no, leave note in PIA details tab: "Owner unknown, redirected to LO tray" and redirect to LO tray for BTO to action.

COI Check Path

Check COI using Vol/Fol or other title details. Does vendor match the registered proprietor in COI? If no, send Vendor Clarification (see Q1). If yes, check: is the vendor the registered proprietor on or before 31 December of the current assessment year?

If ON the date: process PCC as is. If BEFORE the date: does the date of possession impact prior assessment years? If no, process PCC as is. If yes, check if authorised for LO updates. If yes, continue to "Processing from LO tray." If no, redirect to LO tray for BTO action with note "Land Ownership as per dealing number *****."

Processing from the LO Tray (p.10)

BTO and Grade 3 staff authorised to process basic land transfers. Escalations to Assessing are agreed at all times but particularly during busy periods. Using COI, assess the LO update: does it involve a Trust, retro transfers, or multiple parties or payments? If yes, will the LO update impact the assessment? If yes, escalate to Assessing escalation mail box. If no, process PCC as is. If no complex elements, process the LO Transfer as per LO procedures (CM10 file DOC/25/560508) and issue the cert.

Deceased Estate (p.11)

In COI, does the property have dealing type as "transmission application"? If no, check if there is any other information to indicate an estate. If applicants are on COI with transmission application as dealing type, safe to assume they are executors of estate. If no transmission and no other indicators, process PIA as normal.

If estate is detected: determine executors. Check COI. Search Supreme Court (www.supremecourt.vic.gov.au/wills-and-probate), probate records (online.justice.vic.gov.au), Herald Sun tributes (tributes.heraldsun.com.au).

Date of death less than 3 years: load deceased customer to tree, click Edit, enter date of death (DOD) in required field. In comments field enter "DOD as per [source: probate website, Infotrack, Landata, SAI global, LPR etc.]" Save. In PIA application place comment "Trust not created. Vendor is exec." Save and process PIA as normal.

Date of death more than 3 years: Is the estate already a Trust (customer type)? If no, redirect to Estate tray for escalation to ERC with note "DOD as per Probate for PIA Number *****." If yes, CSO to process and issue certificate. In PIA details tab note: "Date of Death as per... Date of death. User ID and date." Redirect PIA to Estate tray.

Source: Change of Ownership Required (p.10); Deceased Estate (p.11)
Input data:
  • e-Sys ownership records (including Owner Unknown status)
  • COI title information (Vol/Fol, dealing types, registered proprietor)
  • Date of possession relative to 31 December assessment date
  • Deceased estate indicators (transmission application dealing type)
  • Date of death (from Supreme Court, probate, Herald Sun, SAI Global)
  • Trust customer type status
  • LO authorisation level (BTO/Grade 3 vs standard CSO)
Processing responsibility:
  • CSO - standard ownership verification via COI
  • BTO / Grade 3 - LO tray processing (basic transfers)
  • Assessing - escalation for trusts, retro transfers, multiple parties
  • Estate tray / ERC - deceased estate processing for DOD over 3 years
  • COI - title verification, dealing type checks
  • Supreme Court / probate websites - date of death confirmation
Decision: Ownership status (four paths):
  • Owner matches and is current: proceed to Q5
  • Owner Unknown: redirect to LO tray (cannot issue)
  • Owner mismatch: Vendor Clarification or LO update required
  • Deceased estate (DOD under 3 years): enter DOD, process as normal
  • Deceased estate (DOD over 3 years): redirect to Estate tray / ERC
Output:
  • Ownership confirmed - proceed
  • Or LO update processed (basic transfer)
  • Or redirected to LO tray, Estate tray, or Assessing

Q5 - Are liabilities, payments, and action items resolved?

  • Suspense Payments (p.12): Launch PIA. Confirm no open action items requiring escalation. Check liability node on all: trust/trustees, LTX group members, all joint ownership members. If suspense payment exists and liability is against same customer: apply to oldest liability first. If different customer: establish source, check if transfer is permitted under SRO guidelines.
  • Payment transfers: Go to Allocations node, tick Suspense, apply. Select payment/s, click Transfer. On transfer screen click Destination. If payment goes from individual to JO/group member: check Same Customer box. If to different customer: enter customer ID of other customer.
  • PIA Escalation - Open Action Items (p.13): Check if PIA customer has active action item. If yes, determine if it will amend the customer's liability. Nine types requiring escalation to assessing@sro.vic.gov.au:
    • Land Ownership
    • LTX Trust Form 13/14
    • PPR
    • PPL
    • Multi Amendments
    • AOS work item
    • Penalty work item
    • Other Amendment
    • Unprocessed NOA (to NOA.Escalation@sro.vic.gov.au)
  • Note: General Correspondence, Complex Correspondence, Formal/Private Rulings do NOT need escalation. If property is in correct ownership, issue certificate as is.
  • Pending Write Off (p.14): Error "A non-issue indicator (Recovery matter specific action) is set for liable party." Check property for liability, then parent, then related properties. If liability ceased before 31 Dec prior year: manual certificate required. If no liabilities on related properties: override error and issue.
  • DMS Recovery Non-Issue (p.15): Same error pattern as Pending Write Off. Check property, parent, related properties for liabilities. Manual certificate path or override path.
  • Vendor Protected by Prior Certificate (p.18): When vendor is protected from prior year liabilities by a previously issued PCC. Confirm original certificate and payment. Enter specific comments in PIA details tab.
  • Zero Value on Property (p.19): Warning that zero-value properties exist. If vendor is statutory body (e.g. Director of Housing): override. If not sure: see Senior CSO. If not statutory: load vendor properties, filter for zero site value, check results.
Action items that will amend liabilities MUST be escalated before issuing certificate.

Suspense Payments (p.12)

Launch PIA work item. Confirm there are no open action items that require escalation. Check liability node on all: trust/trustees, LTX group members, all joint ownership members. Go to Ownership node and load all current owners to the tree including trust/trustees, LTX group members, all joint ownership members. Click on Land Taxpayer node and expand. Click on Allocations and tick Suspense box and click Apply. Did any payments come up? If no, continue on to next customer until suspense payment/s found. If payments found but no outstanding liability, process PIA.

Payment Transfer Process

Is the liability against the same customer with the suspense payment? If same customer: apply suspense payment to oldest liability first. Go to allocations node, tick suspense box and apply. Select payment/s and click Transfer. On transfer payments screen click Destination. Check the Same Customer box. If different customer: was payment actually made by customer with the liability? If yes, apply to oldest liability. If no, establish source of payment. Can payment be transferred to other customer within SRO guidelines? If yes, apply suspense and transfer. If no, send email to relevant staff member to issue certificate.

Transfer Steps in e-Sys

Select the first line of Liability Destination, right-click and select Add. Enter the relevant assessment year into the start and end field. Apply/select fee. Click OK. Liability info will now be populated. In the liability components section tick the box next to the liability. Enter the amount to be used in the transfer amount section. Process PIA.

PIA Escalation - Open Action Items (p.13)

Does PIA customer have an active action item? If no, process PIA as normal. If yes, will the action item amend the customer's liability? Action items to be escalated: Land Ownership, LTX Trust Form 13/14, PPR, PPL, Multi Amendments, Related Party Automation, Other amendment, Penalty work item, AOS work item.

Send email to the relevant team's Escalation Inbox requesting escalation of action item. Email subject: PIA (PIA number) - (Action Item type) - (Date received). In body include: Customer number, Action item type, PIA ID. Put note on PIA details tab "Action item to be completed - user ID." Reserve PIA to your tray. When action item is completed an email will be sent to you. Process PIA and issue as per email.

Recovery Non-Issue / Pending Write Off (p.14-15)

Error: "A non-issue indicator (Recovery matter specific action) is set for liable party." The main reason a write-off is requested is where the liability is less than $50 and it is not economically viable to chase the debt. Usually requested by staff in Debt Management or Land Tax Assessing but may also be placed on the customer by e-Sys.

Check property for a liability. If no, check parent property for liability. If no liability on parent either, refer to Process Online Application. If yes (liability on property or parent), go to related properties node and load parent and grandparent properties to the tree. Is there outstanding liabilities on related properties? If no, override the error and issue certificate. If yes, manual certificate required. Place comment in details tab "Manual certificate required as LTX outstanding on parent property LID xxxxx." Redirect PIA to Manual Certificate Tray.

Vendor Protected by Prior Certificate (p.18)

There are times when the vendor is protected from prior year liabilities due to the property clearance certificate they requested when they purchased the land. Customer contacts to advise prior certificate liability and payment. Request copy of original certificate and payment details. Was original certificate provided? If no, include prior years and issue. If yes, confirm details on certificate. Was payment received for certificate? If no, include prior years and issue. If yes, in details tab of PIA enter: "(original cert number) issued protects vendor from prior year tax for xxxx year/s. As a result xxxx year/s tax not shown on this certificate." Parcels (p.20)

A parcel is a folder created where multiple properties are rated under the one Municipality Property Number (MPN). Is property in a parcel? If yes: go to parcel node, double-click to load parcel to tree. Under PIA node, link ONLY the parcel to the PIA application (do NOT link individual properties). Manual PIA Process (p.21)

The manual process to issue a certificate where there are net liabilities on inactive parent properties is almost exactly the same as processing an ordinary electronic PIA, including issuing electronically in the hourly batches, and being able to apply electronically for updates.

However, they will not automate. Manual intervention is required. Only some users will have the required e-Sys profile (LTX-A_SPWF) and each pro rata calculation must be done separately.

The user will have the capacity to associate multiple parent properties and perform several pro rata calculations to issue an accurate Certificate. The Certificate will display the child liability and the child's portion of the parent and/or grandparent liabilities separately on the certificate. As is currently the case the liability fields can be overwritten where the information on the certificate needs to be different than e-Sys.

Procedure Summary

  • The PCC details the amount of tax secured by a charge on the land, including current year, prior years, WGT, CIPT, VRLT penalty tax, and deferral interest.
  • Approximately 300,000 applications received per year via e-Business system.
  • Applicants: owner, purchaser, or mortgagee of land. Sellers required to obtain PCC as part of Section 32 statement.
  • Fee: $18.90 (increases at 1.16 fee units on 1 July each year).
  • Turnaround: 10 working days. Auto-issues within 24 hours if no warnings.
  • Unlimited reissues for 90 days post-issuance.
  • Time standard: 6 minutes per work item.
  • Key principle: certificate reflects "right person / right property."
  • A PCC is NOT a Land Tax Assessment - it is a snapshot of liabilities at time of issue.

Structural Limitations and Ambiguities

Warning matrix complexity: 12+ warning types each with distinct resolution flowcharts. Some CSO-resolvable, some require Senior CSO, some redirect to other trays. Decision tree is the most complex in any LRA procedure.
Pro rata variants: Area, Unit Entitlement, and 60/40 rule each have conditional triggers and cannot be expressed as simple binary decisions.
Ownership and estate handling: Four distinct paths (Owner Unknown, COI mismatch, date-based, deceased estate) with different tray routing and authorisation requirements.
Liability sub-processes: Four distinct sub-processes (suspense, escalation, recovery NII, vendor protection) collapsed into one procedural step.
Tray routing: Five different processing trays (LO, Estate, Manual Certificate, Cancellation, Land Taxpayer PIA) with different ownership and authorisation rules.
