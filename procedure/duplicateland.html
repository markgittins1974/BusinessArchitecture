Q1 - Has the property been correctly flagged as a duplicate?

  • Receive Request: LGAT receives a Decommission Property Request work item flagging a property as a potential duplicate.
  • Identify Valid Property: Usually, the property with the most history and earlier taxable date is retained as the valid property.
  • Validate References: Confirm references match the valid property using COI, Land Channel (Lassi), and Municipal Revaluation Files.
  • Decision: Is the property correctly flagged as a duplicate?
Do NOT proceed with decommissioning if the property is not a true duplicate. Escalate if uncertain.

Duplicate vs Unrelated Property

Duplicate indicators: Same lot and plan number, same address (or minor variations), overlapping valuation dates, same MPN, same Torrens title.

Unrelated indicators: Different lot/plan numbers, completely different addresses, non-overlapping valuation periods, different corporate owners, different municipality.

Validation Sources

COI (Compliance Operational Intelligence): Check corporate ownership and relationships. Do the two flagged properties have the same owner/structure?

Land Channel (Lassi): Verify property address and lot/plan numbers. Do references match?

Municipal Revaluation Files: Access in e-Sys Tools tab. Check council valuation data for the property.

Decision Rule (GAP)

CRITICAL GAP: Neither document defines what constitutes a "match" or how to handle conflicting validation sources. Escalate if uncertain.

Q2 - Should the Decommission Property Request be rejected?

  • Assess Request: Determine whether the flagged property should be decommissioned or if the request should be rejected.
  • Reject If: References do NOT match the valid property, property is unrelated, or you cannot validate references.
  • Record Rejection: Document why the request is being rejected in the Rejection Comments field.
  • Unlink Property: Uncheck the decommission flag in e-Sys.
  • Close Work Item: Submit rejection — work item automatically closes.
If rejecting: Record detailed comments explaining why property is not a duplicate. Work item closes but property remains active in e-Sys.

Procedure Detail: Rejection vs Acceptance▼

When to REJECT

  • References do not match the valid property
  • Property is unrelated (added in error, not a duplicate)
  • Cannot validate references due to missing or conflicting data
  • Property has active liabilities or recent transactions

When to ACCEPT (Proceed to Q3)

  • References clearly match the valid property
  • External validation (COI, Lassi, Revaluation) confirms duplicate
  • No active liabilities or recent transactions
  • Assessment cycle allows decommissioning

Rejection Comment Examples

  • "References do not match - different lot/plan numbers"
  • "Property has active liabilities as of current assessment year"
  • "COI shows different corporate owner - unrelated property"
Source: Request Rejection Procedures
Input data: Reference validation results from Q1, property relationship details
Processing responsibility: Assessor (decision), e-Sys (flag management)
Decision: Reject request (YES) or proceed to decommission (NO)
Output: Request rejected with comments recorded, OR request retained and proceeds to Q3

Q3 - Can the duplicate property be decommissioned?

  • Check Authorization: You must have appropriate e-Sys profile to decommission properties. If not, escalate to senior officer.
  • Clear Information: Remove all identifying information from the duplicate property (address, title, municipal, planning, area fields).
  • Add End Date: In the "Other" tab, enter End Date: 29/01/[current year]
  • Record Comments: Document why the property is being decommissioned (e.g., "Duplicate of LOT 100 PS 123456").
  • Save Changes: Click Save to commit decommission. Work item automatically completes.
Only Grade 3 and above can decommission. If you lack permission, contact senior officer immediately.

Critical Gaps Identified

GAP 1 (HIGH): Duplicate vs Unrelated Property — No clear decision rule to distinguish. Staff may incorrectly decommission valid properties. Implement decision matrix with duplicate/unrelated indicators.
GAP 2 (HIGH): Valid Property Selection Criteria — Rule "property with most history and earliest taxable date is retained" is mentioned but not formalized. No fallback rule if dates are equal. Create formal decision hierarchy.
GAP 3 (HIGH): Reference Validation Methodology — No guidance on HOW to use COI, Lassi, and Revaluation Files. What constitutes a "match"? What if sources conflict? Validation is subjective. Create validation protocol.
GAP 4 (MEDIUM): End Date Determination — LGA shows "29/01/xx" but does not explain why. Is it assessment year end? Discovery date? Clarify end date rule.
GAP 5 (MEDIUM): Authorization & Escalation — No clear path if assessor lacks decommission authority. Define authorization matrix (who can reject, who can decommission, who is senior officer).
GAP 6 (MEDIUM): Work Item Lifecycle — No intermediate states or suspension procedures documented. Define states: received → validating → awaiting auth → rejected/decommissioned.
GAP 7 (MEDIUM): Request Rejection vs Property Decommission — Terminology is ambiguous. Clarify: Rejection closes work item; Decommission removes property. Are they separate outcomes?
GAP 8 (MEDIUM): LGA Document Incomplete — Purpose section has template placeholders ("Click or tap..."). Revision date not filled in. Complete the document.

Frequently Asked Questions

Q: What's the difference between a duplicate property and an unrelated property?

A: A duplicate is the same property recorded twice (same lot/plan, same address). An unrelated property is a completely different property added in error. Use COI, Lassi, and Revaluation data to distinguish. GAP: No formal decision criteria provided.

Q: Which property do I keep if both have the same creation date?

A: The procedure mentions "most history and earlier taxable date" but this needs clarification. If dates and history are equal, escalate to Grade 3 for decision. GAP: Fallback rule missing.

Q: What if COI and Lassi show different information?

A: Document the conflict and escalate to Grade 3 for decision. Do not proceed without resolution. GAP: Conflict resolution procedure missing.

Q: Can I decommission a property with active liabilities?

A: No. If the property has open assessments, recent transactions, or unpaid liabilities, reject the request and escalate to Grade 3.

Q: What happens if I reject the request?

A: The work item closes, but the flagged property remains in e-Sys active. It is not deleted; it remains for further investigation.

Q: What if I don't have authorization to decommission?

A: Contact your Grade 3 supervisor or senior officer. Request they complete the decommissioning. Work item will remain open until they complete it. GAP: Escalation SLA not defined.

Q: What end date should I use?

A: LGA example shows "29/01/xx" but significance is unclear. Use decommission discovery date or assessment year end, and document your choice in comments. GAP: End date rule undefined.

Q: Can I undo a decommission?

A: Procedure does not cover reverting changes. If you made an error, contact Grade 3 or 4 immediately. Do not attempt to modify without approval.
