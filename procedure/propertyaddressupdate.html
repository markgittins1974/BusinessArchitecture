Property Address Update Procedure for e-Sys

Division: Land Revenue • Branch: Operations • Revised: 1 July 2021

Q1 Source of Advice Q2 Property Identification Q3 Address Verification Q4 Address Update
```

Q1 - What is the source of the address change advice?

  • Council-provided advice: Councils send address change notifications via email or mail to the LGA Team. These typically include lot and plan number, MPN, old address, and new address.
  • Customer-provided advice: Customers may contact the SRO by phone, email, or letter to advise that the property address on their land tax assessment is incorrect.
  • Action: Receive and classify the advice source (Council or Customer).
  • Next step: Proceed to Q2 (Property Identification) for the identified source path.
Processing responsibility: LGA Team receives council advice; Operations staff handles customer enquiries.
Procedure reference: Background; Council Provided Information; Customer Provided Information
Procedure Detail: Advice Sources▼

Council-Provided Information

Councils send advice of a change in a property address either via email or mail. The advice is distributed to a member of the LGA Team for processing. The council advice typically provides: lot and plan number, MPN, old address, and new address.

Customer-Provided Information

A customer may ring the SRO, send an email or letter to advise a property address on their land tax assessment is incorrect. The customer may have identified that the address on their council rates notice differs from the address on their land tax assessment, or they may simply know the property by a different address.

Processing Assignment

Council advice is assigned to the LGA Team. Customer advice is handled by Operations staff who must investigate whether the customer has identified the correct property and determine if address verification is required.

Source: Background; Council Provided Information; Customer Provided Information sections
Input data: Address change advice (letter, phone, email)
Processing responsibility: LGA Team (council advice) or Operations staff (customer advice)
Decision: Source classification (Council / Customer)
Output: Processing path determined (council path = direct update; customer path = verification required)

Q2 - Can the correct property be identified in e-Sys?

  • Identification method: Use the lot and plan number (e.g., lot 1 PS 715854) to search for the property in e-Sys.
  • Alternative: If lot and plan is unavailable, use MPN (Municipal Property Number) to locate the property.
  • Verification: Confirm the property record found corresponds to the details in the address change advice.
  • Council path: If correct property identified, proceed to Q4 (Address Update). Council-provided data is accepted as verified.
  • Customer path: If correct property identified, proceed to Q3 (Address Verification). Customer-provided data requires additional verification steps.
  • Escalation: If you suspect an incorrect property has been included, escalate to Grade 3 or 4 for advice before proceeding.
Do not proceed with address update if the correct property cannot be identified. Escalate to Grade 3 or 4.
Procedure reference: Council Provided Information; Customer Provided Information
Procedure Detail: Property Identification▼

Search Method

Use the lot and plan number to search for the property in e-Sys. The example provided shows searching for "lot 1 PS 715854." This combination of lot number and plan reference uniquely identifies Victorian land parcels.

Confirmation Step

Once you have located a property record matching the lot and plan number, confirm the property record corresponds to the details provided in the address change advice. For council advice, this may simply be confirming the address details match. For customer advice, you must verify the property is correct before proceeding to address verification.

Escalation Trigger

If you suspect an incorrect property has been included in the address change notification (for example, if lot and plan searches return no results, or if the property details do not match the advice), speak to your Grade 3 or 4 for advice before proceeding.

Source: Council Provided Information and Customer Provided Information sections
Input data: Lot and plan number, MPN, address details from advice
Processing responsibility: e-Sys (property search and lookup)
Decision: Correct property identified (Yes / No) or Escalation required
Output: Property record selected for update, or escalation to Grade 3/4 initiated

Q3 - Is the address change supported by authoritative data?

  • Customer path only: This question applies only to customer-provided address changes. Council-provided addresses proceed directly to Q4.
  • Step 1 - Municipal Revaluation Files: Access the "View Municipal Revaluation" files within e-Sys (Tools tab). Check if the files reflect the address details the customer is advising.
  • Step 2 - Land Channel (Lassi): If municipal revaluation files do not support the customer's claim, refer to Land Channel (Lassi) via the intranet for the property address. Lassi is a reliable source because councils regularly update details, and Lassi is typically updated before supplementary valuations are processed by SRO.
  • Step 3 - Rates Notice: If you are unable to confirm address details via either municipal revaluation files or Lassi, request the customer send in a copy of their current council rates notice. Ensure the rates notice is current and relates to the property found in Q2.
  • Confirmation rule: Do not proceed with address update if address cannot be confirmed by at least one authoritative source.
  • Council communication: When the address differs from the address advised by the customer, the address should be confirmed with council. Speak to the Grade 3 responsible for contacting council.
Do not proceed if address cannot be confirmed against authoritative data. Address must be verified before update.
Procedure reference: Customer Provided Information; verification sources
Procedure Detail: Verification Hierarchy▼

Verification Source Priority

The procedure establishes a verification hierarchy for customer-provided address changes:

  1. Municipal Revaluation Files (Primary): Within e-Sys, access Tools tab > View Municipal Revaluation. This reflects recent council data submissions for annual and supplementary valuations.
  2. Land Channel / Lassi (Secondary): If revaluation files do not support the customer claim, check Lassi. Lassi is generally updated faster than SRO processes supplementary data, making it a reliable secondary source.
  3. Council Rates Notice (Tertiary): If both above sources are inconclusive, request a current rates notice from the customer. This must be the current rates notice for the property identified in Q2.

Conflicting Data Rule

When the address differs from what the customer advised, the address should be confirmed with council before proceeding. The Grade 3 responsible for council liaison should be contacted.

Non-Confirmation Rule

Do not proceed with address update if address cannot be confirmed via at least one authoritative source. This prevents unverified customer claims from being recorded in the official address register.

Source: Customer Provided Information section
Input data: Municipal revaluation data, Lassi data, rates notice (if requested), customer's claimed address
Processing responsibility: e-Sys (revaluation files), Land Channel / Lassi (address verification), Operations staff (rates notice request and council liaison)
Decision: Address verified against authoritative source (Confirmed / Not confirmed / Requires council verification)
Output: Address confirmed for update, or escalation to council contact required, or customer request denied
Gate rule: Customer-provided addresses cannot proceed past Q3 without verification confirmation.

Q4 - Can the address be updated?

  • Edit the property record: Once the correct property is identified (Q2) and address is verified (Q3 for customer path), select the Edit button at the top right hand side of the e-Sys property screen.
  • Update the address: In the property details, change the address from the old address to the new address as advised or verified. Example: change from "6 Durville Street" to "6B Durville Street".
  • Save the update: Select the Save icon at the top right hand side of the screen to save the updated property address.
  • Completion: Once saved, the property address record has been updated in e-Sys and the address change process is complete.
Procedure reference: Council Provided Information (Address Update Steps); Customer Provided Information (Post-Verification Update)
Procedure Detail: Update Steps▼

Accessing Edit Mode

Load the property record identified in Q2. Select the Edit button located at the top right hand side of the property screen. This enables edit mode for the property details.

Address Field Update

Locate the address field in the property details. Replace the old address with the new address as provided in the council advice or verified through the customer verification process. The example shows updating "6 Durville Street" to "6B Durville Street".

Save Action

After entering the new address, select the Save icon at the top right hand side of the screen. This commits the address change to e-Sys and completes the property address update process.

Post-Update Verification

After saving, the property record should reflect the updated address. No further steps are required unless address-related assessments or correspondence need to be regenerated (which is outside the scope of this procedure).

Source: Council Provided Information and Customer Provided Information sections
Input data: Verified address details, property record selected in Q2
Processing responsibility: e-Sys (edit and save property record)
Decision: Update address field in property record
Output: Property address updated and saved in e-Sys database

Procedure Summary

  • Establishes the steps for updating property address details in e-Sys when advised by councils or customers.
  • Covers two intake paths: council-provided advice and customer-provided advice.
  • Requires identification of the correct property using lot and plan numbers, MPN, and valuation data.
  • Includes verification of address accuracy using municipal revaluation files, Land Channel (Lassi), and rate notices.
  • Ensures property address records reflect verified and authoritative data prior to update.
  • Council-provided addresses bypass verification; customer-provided addresses require verification before update.

Decision Framework

Question Decision Point Yes Path No Path
Q1 What is the source? Classify as Council or Customer → proceed to Q2
Q2 Property identified? Proceed to Q3/Q4 (based on source) Escalate to Grade 3/4; do not proceed
Q3 Address verified? (Customer only) Proceed to Q4 Contact council or deny request; do not proceed
Q4 Update address? Edit and Save address in e-Sys → Complete

Verification Hierarchy (Customer Path Q3)

  1. Primary: Municipal Revaluation Files (e-Sys Tools tab)
  2. Secondary: Land Channel (Lassi) intranet
  3. Tertiary: Current council rates notice (request from customer if above inconclusive)
  4. If conflicting: Escalate to council liaison (Grade 3)

Key Process Paths

  • Council Path (Q1→Q2→Q4): Address change from council → Identify property → Direct update (no verification required)
  • Customer Path (Q1→Q2→Q3→Q4): Address correction from customer → Identify property → Verify address → Update if confirmed
  • Escalation (any Q): If property cannot be identified (Q2) or address cannot be verified (Q3), escalate to Grade 3 or 4

Critical Gaps Identified

Gap 1 (HIGH): No explicit verification thresholds for customer data. Procedure does not define when conflicting sources require council confirmation vs when one source is sufficient.
Gap 2 (HIGH): Escalation triggers undefined. No decision criteria for when to escalate (e.g., if lot/plan search returns zero results, how many results trigger escalation?).
Gap 3 (MEDIUM): Council data acceptance rule not explicit. Procedure implies council-provided data is accepted as-is but does not state this explicitly or explain why.
Gap 4 (MEDIUM): Missing error handling. No procedures for when e-Sys is unavailable, Lassi cannot be accessed, or property search returns multiple matches.
Gap 5 (MEDIUM): No audit trail or record-keeping requirements. Procedure does not specify whether address changes are logged or what evidence of verification should be retained.
```
